# 코딩 에이전트는 어떻게 작동하는가

어떤 도구와 마찬가지로, [코딩 에이전트](https://simonwillison.net/guides/agentic-engineering-patterns/what-is-agentic-engineering/)가 내부적으로 어떻게 작동하는지 이해하면 이를 적용하는 방법에 대해 더 나은 결정을 내리는 데 도움이 될 수 있습니다.

코딩 에이전트는 LLM을 위한 하네스(harness) 역할을 하는 소프트웨어 조각으로, 보이지 않는 프롬프트에 의해 구동되고 호출 가능한 도구로 구현된 추가 기능으로 해당 LLM을 확장합니다.

## 대규모 언어 모델 (Large Language Models)

모든 코딩 에이전트의 중심에는 대규모 언어 모델(LLM)이 있습니다. 이들은 GPT-5.4, Claude Opus 4.6, Gemini 3.1 Pro 또는 Qwen3.5-35B-A3B와 같은 이름을 가지고 있습니다.

LLM은 텍스트 문장을 완성할 수 있는 기계 학습 모델입니다. 모델에 "the cat sat on the "라는 구를 주면 (거의 확실하게) 문장의 다음 단어로 "mat"을 제안할 것입니다.

이러한 모델이 더 커지고 더 많은 양의 데이터로 훈련됨에 따라, "URL에서 파일을 다운로드하는 파이썬 함수는 def download_file(url): "과 같은 더 복잡한 문장을 완성할 수 있습니다.

LLM은 실제로 단어와 직접 작동하지 않고 토큰과 함께 작동합니다. 텍스트 시퀀스는 정수 토큰 시퀀스로 변환되므로 "the cat sat on the "는 `[3086, 9059, 10139, 402, 290, 220]`이 됩니다. LLM 제공업체는 처리된 토큰 수를 기준으로 요금을 청구하고 한 번에 고려할 수 있는 토큰 수에 제한이 있기 때문에 이는 이해할 가치가 있습니다.

OpenAI 토크나이저를 [platform.openai.com/tokenizer](https://platform.openai.com/tokenizer)에서 실험하여 이것이 어떻게 작동하는지 확인할 수 있습니다.

LLM에 대한 입력을 프롬프트라고 합니다. LLM이 반환하는 텍스트를 완료(completion) 또는 때로는 응답(response)이라고 합니다.

오늘날 많은 모델은 다중 모달(multimodal)이므로 텍스트 이상의 것을 입력으로 받아들일 수 있습니다. 비전 LLM(vLLM)은 입력의 일부로 이미지를 받아들일 수 있으므로 스케치, 사진 또는 스크린샷을 제공할 수 있습니다. 일반적인 오해는 이것이 OCR 또는 이미지 분석을 위해 별도의 프로세스를 통해 실행된다는 것이지만, 이러한 입력은 실제로는 텍스트와 동일한 방식으로 처리되는 더 많은 토큰 정수로 변환됩니다.

## 채팅 템플릿 프롬프트 (Chat templated prompts)

최초의 LLM은 완료 엔진으로 작동했습니다. 사용자는 위에서 보여준 두 가지 예와 같이 모델에 의해 완료될 수 있는 프롬프트를 제공해야 했습니다.

이는 특히 사용자 친화적이지 않았기 때문에 모델은 대부분 대신 채팅 템플릿 프롬프트를 사용하는 것으로 전환했으며, 이는 모델과의 통신을 시뮬레이션된 대화로 나타냅니다.

이것은 실제로는 다음과 같은 특수 형식을 가진 완료 프롬프트의 한 형태일 뿐입니다.

```
user: URL에서 파일을 다운로드하는 파이썬 함수를 작성해 줘
assistant:
```

이 프롬프트에 대한 자연스러운 완료는 (LLM으로 대표되는) 어시스턴트가 일부 파이썬 코드로 사용자의 질문에 답하는 것입니다.

LLM은 상태 비저장(stateless)입니다. 프롬프트를 실행할 때마다 동일한 빈 상태에서 시작합니다.

대화 시뮬레이션을 유지하기 위해 모델과 통신하는 소프트웨어는 자체 상태를 유지하고 사용자가 새 채팅 프롬프트를 입력할 때마다 전체 기존 대화를 다시 재생해야 합니다.

```
user: URL에서 파일을 다운로드하는 파이썬 함수를 작성해 줘
assistant: def download_url(url):
    return urllib.request.urlopen(url).read()
user: 대신 requests 라이브러리를 사용해 줘
assistant:
```

제공업체는 입력 및 출력 토큰 모두에 대해 요금을 청구하므로 대화가 길어질수록 입력 토큰 수가 매번 증가하므로 각 프롬프트가 더 비싸집니다.

## 토큰 캐싱 (Token caching)

대부분의 모델 제공업체는 캐시된 입력 토큰에 대해 더 저렴한 요금을 통해 이를 다소 상쇄합니다. 짧은 시간 내에 처리된 공통 토큰 접두사는 기본 인프라가 해당 입력을 처리하는 데 사용되는 많은 비용이 많이 드는 계산을 캐시하고 재사용할 수 있으므로 더 낮은 요금으로 청구될 수 있습니다.

코딩 에이전트는 이러한 최적화를 염두에 두고 설계되었습니다. 캐시가 가능한 한 효율적으로 사용되도록 이전 대화 내용을 수정하는 것을 피합니다.

## 도구 호출 (Calling tools)

LLM 에이전트의 결정적인 특징은 에이전트가 도구를 호출할 수 있다는 것입니다. 하지만 도구란 무엇일까요?

도구는 에이전트 하네스가 LLM에 제공하는 함수입니다.

프롬프트 자체 수준에서는 다음과 같이 보입니다.

```
system: 날씨에 접근해야 하는 경우 <tool>get_weather(city_name)</tool>으로 차례를 마치세요.
user: 샌프란시스코 날씨는 어때?
assistant:
```

여기서 어시스턴트는 다음 텍스트로 응답할 수 있습니다.

```
<tool>get_weather("San Francisco")</tool>
```

그런 다음 모델 하네스 소프트웨어는 응답에서 해당 함수 호출 요청을 추출하고(아마도 정규식으로) 도구를 실행합니다.

그런 다음 다음과 같이 구성된 프롬프트와 함께 결과를 모델에 반환합니다.

```
system: 날씨에 접근해야 하는 경우 <tool>get_weather(city_name)</tool>으로 차례를 마치세요.
user: 샌프란시스코 날씨는 어때?
assistant: <tool>get_weather("San Francisco")</tool>
user: <tool-result>61°, Partly cloudy</tool-result>
assistant:
```

이제 LLM은 해당 도구 결과를 사용하여 사용자의 질문에 대한 답변을 생성하는 데 도움을 줄 수 있습니다.

대부분의 코딩 에이전트는 에이전트가 호출할 수 있는 12개 이상의 도구를 정의합니다. 이 중 가장 강력한 것은 코드 실행을 허용하는 것입니다. 예를 들어 터미널 명령을 실행하기 위한 `Bash()` 도구나 파이썬 코드를 실행하기 위한 `Python()` 도구가 있습니다.

## 시스템 프롬프트 (The system prompt)

이전 예에서는 사용 가능한 도구와 호출 방법에 대해 LLM에 알리는 "system"으로 표시된 초기 메시지를 포함했습니다.

코딩 에이전트는 일반적으로 사용자에게 표시되지 않지만 모델이 어떻게 작동해야 하는지에 대한 지침을 제공하는 이러한 시스템 프롬프트로 모든 대화를 시작합니다.

이러한 시스템 프롬프트는 수백 줄에 달할 수 있습니다. 2026년 3월 현재 [OpenAI Codex의 시스템 프롬프트](https://gist.github.com/simonw/17391888e3c651a24e4d5b312368133b)는 이러한 코딩 에이전트가 작동하도록 만드는 지침의 종류에 대한 유용하고 명확한 예입니다.

## 추론 (Reasoning)

2025년의 큰 새로운 발전 중 하나는 프론티어 모델 제품군에 추론이 도입된 것입니다.

추론은 때때로 UI에서 생각하는 것으로 표시되며, 모델이 사용자에게 회신을 제시하기 전에 문제를 해결하고 잠재적인 해결책을 설명하는 텍스트를 생성하는 데 추가 시간을 소비하는 경우입니다.

이것은 사람이 소리 내어 생각하는 것과 비슷하게 보일 수 있으며 비슷한 효과를 가집니다. 결정적으로 모델이 더 나은 결과를 얻기 위해 문제에 더 많은 시간(그리고 더 많은 토큰)을 소비할 수 있도록 합니다.

추론은 모델이 더 복잡한 코드 경로를 탐색하고, 도구 호출을 혼합하고, 추론 단계를 사용하여 함수 호출을 문제의 잠재적 원인으로 다시 추적할 수 있는 기회를 제공하므로 코드의 문제를 디버깅하는 데 특히 유용합니다.

많은 코딩 에이전트에는 추론 노력 수준을 높이거나 낮추는 옵션이 포함되어 있어 모델이 더 어려운 문제를 해결하는 데 더 많은 시간을 할애하도록 권장합니다.

## LLM + 시스템 프롬프트 + 루프 안의 도구

믿거나 말거나, 이것이 코딩 에이전트를 구축하는 데 필요한 대부분입니다!

이러한 것들이 어떻게 작동하는지에 대한 더 깊은 이해를 개발하고 싶다면, 처음부터 자신만의 에이전트를 구축해 보는 것이 유용한 연습입니다. 간단한 도구 루프는 기존 LLM API 위에 수십 줄의 코드로 달성할 수 있습니다.

좋은 도구 루프는 그보다 훨씬 더 많은 작업이지만, 기본적인 메커니즘은 놀라울 정도로 간단합니다.
