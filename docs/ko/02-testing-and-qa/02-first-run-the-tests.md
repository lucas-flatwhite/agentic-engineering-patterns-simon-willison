# 먼저 테스트를 실행하라

코딩 에이전트와 함께 작업할 때 자동화 테스트는 더 이상 선택 사항이 아닙니다.

테스트를 안 쓰던 과거의 변명들, 즉 코드베이스가 빠르게 진화하는 동안 테스트를 지속적으로 다시 쓰는 일이 시간도 많이 들고 비용도 크다는 주장은, 이제 에이전트가 몇 분 안에 테스트를 정돈해줄 수 있기 때문에 더 이상 성립하지 않습니다.

또한 자동화 테스트는 AI가 생성한 코드가 실제로 주장하는 대로 동작하는지를 확인하는 데 _핵심적_입니다. 코드가 한 번도 실행되지 않았다면, 프로덕션에 배포했을 때 동작하는지는 사실상 운에 맡기는 셈입니다.

테스트는 에이전트가 기존 코드베이스에 빠르게 적응하도록 돕는 훌륭한 도구이기도 합니다. 기존 기능에 대해 Claude Code 같은 도구에 물어보면 어떻게 되는지 보세요. 높은 확률로 관련 테스트를 찾아 읽습니다.

에이전트는 이미 테스트 쪽으로 편향되어 있지만, 기존 테스트 스위트가 존재하면 자신이 만드는 새 변경에 대해서도 테스트를 실행할 가능성이 거의 확실히 높아집니다.

기존 프로젝트에서 에이전트로 새 세션을 시작할 때마다 저는 보통 아래와 비슷한 프롬프트로 시작합니다:

```
First run the tests
```

제 Python 프로젝트에서는 [pyproject.toml 설정](https://til.simonwillison.net/uv/dependency-groups)을 해두어서 대신 이렇게 프롬프트할 수 있습니다:

```
Run "uv run pytest"
```

이 네 단어짜리 프롬프트는 여러 목적을 동시에 수행합니다:

1.  테스트 스위트가 존재한다는 사실을 에이전트에 알려주고, 테스트 실행 방법을 스스로 파악하게 만듭니다. 이렇게 하면 이후에도 무언가를 망가뜨리지 않았는지 확인하기 위해 테스트를 실행할 가능성이 매우 높아집니다.
2.  대부분의 테스트 하니스는 테스트 개수의 대략적인 규모를 알려줍니다. 이는 프로젝트의 크기와 복잡도를 가늠하는 지표 역할을 하며, 더 알고 싶다면 테스트 자체를 탐색해야 한다는 힌트도 줍니다.
3.  에이전트를 테스트 중심 사고방식으로 전환시킵니다. 한 번 테스트를 실행하고 나면, 이후 스스로 테스트를 확장하는 행동으로 자연스럽게 이어집니다.

["Use red/green TDD"](https://simonwillison.net/guides/agentic-engineering-patterns/red-green-tdd/)와 마찬가지로, "First run the tests" 역시 모델에 이미 내재된 상당한 소프트웨어 엔지니어링 규율을 압축해서 전달하는 네 단어 프롬프트입니다.
