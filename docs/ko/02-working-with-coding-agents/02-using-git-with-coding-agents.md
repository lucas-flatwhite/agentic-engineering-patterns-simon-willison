# 코딩 에이전트와 Git 함께 사용하기

Git은 코딩 에이전트와 작업할 때 핵심적인 도구입니다. 코드를 버전 관리 시스템에 유지하면 시간 경과에 따른 코드 변경 사항을 기록하고, 실수를 조사하고 되돌릴 수 있습니다. 모든 코딩 에이전트는 Git의 기본 및 고급 기능 사용에 능숙합니다.

이러한 능숙함은 우리가 Git을 사용하는 방식에 대해 더 야심 찰 수 있음을 의미합니다. Git으로 무언가를 하는 _방법_을 외울 필요는 없지만, 무엇이 가능한지 알고 있으면 Git의 모든 기능을 활용할 수 있습니다.

## Git 필수 요소

각 Git 프로젝트는 **저장소(repository)**에 존재합니다. 이는 디스크의 폴더로, 내부 파일의 변경 사항을 추적할 수 있습니다. 이러한 변경 사항은 **커밋(commits)**으로 기록됩니다. 커밋은 하나 이상의 파일에 대한 변경 사항의 타임스탬프가 찍힌 묶음으로, 변경 사항을 설명하는 **커밋 메시지(commit message)**와 변경을 기록한 **작성자(author)**가 함께 제공됩니다.

Git은 **브랜치(branches)**를 지원하여 서로 독립적으로 새로운 변경 사항을 구성하고 실험할 수 있습니다. 브랜치는 준비가 되면 (다양한 방법을 사용하여) 메인 브랜치로 다시 **병합(merged)**될 수 있습니다.

Git 저장소는 새 머신으로 **복제(cloned)**될 수 있으며, 이 복제본에는 현재 파일과 전체 변경 내역이 모두 포함됩니다.
이는 개발자나 코딩 에이전트가 추가 네트워크 트래픽 없이 해당 내역을 탐색하고 탐험할 수 있음을 의미하며, 사실상 내역 탐색을 무료로 만듭니다.

Git 저장소는 자신의 머신에만 존재할 수도 있지만, Git은 원격 저장소(**remote**)에 게시하여 협업과 백업을 지원하도록 설계되었습니다. 원격 저장소는 공개 또는 비공개일 수 있습니다. GitHub는 이러한 원격 저장소를 위한 가장 인기 있는 장소이지만, Git은 Git 프로토콜을 지원하는 모든 머신이나 서비스에서 이러한 원격 저장소를 호스팅할 수 있는 오픈 소스 소프트웨어입니다.

## 핵심 개념과 프롬프트

코딩 에이전트는 모두 Git 전문 용어에 대한 깊은 이해를 가지고 있습니다. 다음 프롬프트는 어떤 에이전트와도 작동할 것입니다:

Start a new Git repo here

에이전트가 작업 중인 폴더를 Git 저장소로 바꾸려면, 에이전트는 아마도 `git init` 명령을 실행할 것입니다. 그냥 "repo"라고 말하면 에이전트는 Git 저장소를 의미한다고 가정할 것입니다.

Commit these changes

에이전트가 만든 변경 사항을 기록하기 위해 새 Git 커밋을 생성합니다. 보통 `git commit -m "commit message"` 명령을 사용합니다.

Add username/repo as a github remote

이것은 GitHub용 저장소를 구성해야 합니다. 먼저 [github.com/new](https://github.com/new)를 사용하여 새 저장소를 만들어야 하고, GitHub와 통신하도록 머신을 구성해야 합니다.

Review changes made today

또는 "recent changes" 또는 "last three commits".

이것은 새로운 코딩 에이전트 세션을 시작하는 좋은 방법입니다. 에이전트에게 최근 변경 사항을 보도록 지시하면 `git log`를 실행하게 되어, 최근에 작업한 내용(수정된 코드와 이를 설명하는 커밋 메시지 모두)에 대한 세부 정보로 컨텍스트를 즉시 로드할 수 있습니다.

이런 식으로 세션을 시딩하면 해당 코드에 대해 이야기하기 시작할 수 있습니다. 추가 수정을 제안하거나, 작동 방식에 대해 질문하거나, 이전에 구축된 것을 기반으로 다음 변경을 제안할 수 있습니다.

Integrate latest changes from main

원격 저장소에서 다른 기여를 가져오려면 메인 브랜치에서 이 명령을 실행하거나, 브랜치에서 실행하여 main의 최신 변경 사항을 통합하세요.

변경 사항을 병합하는 방법에는 merge, rebase, squash 또는 fast-forward 등 여러 가지가 있습니다. 이들의 세부 사항을 기억하지 못해도 괜찮습니다:

Discuss options for integrating changes from main

에이전트는 다양한 병합 전략의 장단점을 설명하는 데 능숙하며, git의 모든 것은 항상 되돌릴 수 있으므로 새로운 것을 시도하는 데 최소한의 위험만 따릅니다.
Sort out this git mess for me

저는 이 보편적인 프롬프트를 놀라울 정도로 자주 사용합니다! 병합 충돌로 실패한 체리픽을 수정한 [최근 예시](https://gisthost.github.io/?2aa2ee2fbd08d272528bbfc3b54a1a7d/page-001.html)가 있습니다.

Git에서는 병합 충돌로 끝나는 pull 또는 rebase 명령을 통하거나, Git의 스테이징 환경에 잘못된 것을 추가하는 등 여러 가지 방법으로 엉망이 될 수 있습니다.

이러한 문제를 해결하는 것은 이전에는 Git 작업에서 가장 어렵고 시간이 많이 걸리는 부분이었습니다. 더 이상은 아닙니다! 코딩 에이전트는 가장 복잡한 병합 충돌을 탐색하고, 새 코드의 의도를 추론하며, 무엇을 유지하고 충돌하는 변경 사항을 어떻게 결합할지 알아낼 수 있습니다. 코드에 자동화된 테스트가 있다면([있어야 합니다](https://simonwillison.net/guides/agentic-engineering-patterns/red-green-tdd/)), 에이전트는 병합을 완료하기 전에 해당 테스트가 통과하는지 확인할 수 있습니다.

Find and recover my code that does ...

이전에 커밋되었거나 `git stash`로 저장된 작업 중인 코드를 잃어버렸다면, 에이전트가 아마 찾아줄 수 있을 것입니다.

Git에는 영구 브랜치에 커밋되지 않은 코드의 세부 정보를 종종 캡처할 수 있는 `reflog`라는 메커니즘이 있습니다. 에이전트는 그것을 검색하고 다른 브랜치도 검색할 수 있습니다.

그냥 무엇을 찾을지 알려주고 그들이 뛰어드는 것을 지켜보세요.

Use git bisect to find when this bug was introduced: ...

Git bisect는 Git의 무기고에서 가장 강력한 디버깅 도구 중 하나이지만, 상대적으로 가파른 학습 곡선 때문에 개발자들이 사용을 꺼리는 경우가 많습니다.

bisect 작업을 실행할 때, Git에 일종의 테스트 조건과 시작 및 종료 커밋 범위를 제공합니다. 그러면 Git은 이진 검색을 실행하여 테스트 조건이 실패하는 가장 빠른 커밋을 식별합니다.

이것은 "이 버그를 처음 유발한 것은 무엇인가"라는 질문에 효율적으로 답할 수 있습니다. 유일한 단점은 Git bisect가 실행할 수 있는 형식으로 버그에 대한 테스트를 표현해야 한다는 것입니다.

코딩 에이전트는 이 상용구 코드를 처리할 수 있습니다. 이것은 Git bisect를 가끔 사용하는 도구에서 소프트웨어의 역사적 동작에 대해 궁금할 때마다 배포할 수 있는 도구로 업그레이드합니다.

## 내역 재작성

재미있는 고급 내용으로 들어가 봅시다.

Git 저장소의 커밋 내역은 고정되어 있지 않습니다. 데이터는 결국 디스크의 파일일 뿐이며(숨겨진 `.git/` 디렉토리에 들어 있음), Git 자체는 해당 내역을 수정하는 데 사용할 수 있는 도구를 제공합니다.

Git 내역을 실제로 일어난 일의 영구적인 기록으로 생각하지 마세요. 대신 소프트웨어 프로젝트의 진행 상황을 설명하는 의도적으로 작성된 이야기로 생각하세요.

이 이야기는 미래 개발을 돕는 도구입니다. 실수와 취소된 방향을 영구적으로 기록하는 것이 때로는 유용할 수 있지만, 저장소 작성자는 무엇을 유지하고 해당 내역을 가장 잘 캡처하는 방법에 대해 편집 결정을 내릴 수 있습니다.

코딩 에이전트는 Git의 고급 내역 재작성 기능을 사용하는 데 정말 능숙합니다.

### 커밋 되돌리기 또는 재작성하기

Undo last commit

코드를 커밋한 다음 후회하는 것은 흔한 일입니다. 예를 들어 포함하고 싶지 않았던 파일이 포함된 것을 깨닫는 경우입니다. 이에 대한 git 레시피는 `git reset --soft HEAD~1`입니다. 저는 이것을 결코 기억할 수 없었고, 이제는 그럴 필요가 없습니다!

Remove uv.lock from that last commit

커밋에 대해 더 세밀한 수술을 수행할 수도 있습니다. 예를 들어 단일 파일을 제거하도록 재작성하는 것입니다.

Combine last three commits with a better commit message

에이전트는 커밋 메시지를 재작성하고 여러 커밋을 단일 단위로 결합할 수 있습니다.

저는 프론티어 모델이 보통 커밋 메시지에 대해 정말 좋은 취향을 가지고 있다는 것을 발견했습니다. 예전에는 직접 작성해야 한다고 주장했지만, 그들이 생산하는 품질이 일반적으로 충분히 좋고, 종종 제가 직접 만들었을 것보다 더 낫다는 것을 받아들였습니다.

### 오래된 저장소의 조각으로 새 저장소 구축하기

제가 꽤 자주 사용하는 트릭은 더 큰 저장소에서 코드를 추출하여 해당 코드의 핵심 내역을 유지하면서 새 저장소로 만드는 것입니다.

한 가지 일반적인 예는 라이브러리 추출입니다. 프로젝트에 일부 클래스와 함수를 구축했다가 나중에 독립적인 재사용 가능한 코드 라이브러리로 만드는 것이 더 합리적이라는 것을 깨달을 수 있습니다.

Start a new repo at /tmp/distance-functions and build a Python library there with the lib/distance_functions.py module from here - build a similar commit history copying the author and commit dates in the new repo

이런 종류의 작업은 대부분의 개발자가 이전 커밋 내역과 분리된 새 복사본을 만들 정도로 관련이 있었습니다. 우리는 더 이상 그것에 만족할 필요가 없습니다!
