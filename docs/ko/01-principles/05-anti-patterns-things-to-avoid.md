# 안티패턴: 피해야 할 것들

에이전틱 엔지니어링이라는 우리의 이상한 신세계에는 안티패턴인 몇 가지 행동들이 있습니다.

## 검토하지 않은 코드를 동료에게 떠넘기기

이 안티패턴은 흔하고 매우 실망스럽습니다.

**직접 검토하지 않은 코드로 풀 리퀘스트를 제출하지 마십시오**.

에이전트가 생성한 수백 (또는 수천) 줄의 코드로 PR을 열고, 그 코드가 기능적으로 작동하는지 확인하는 작업을 직접 하지 않았다면, 당신은 실제 작업을 다른 사람에게 위임하는 것입니다.

그들도 직접 에이전트에게 프롬프트를 보낼 수 있었습니다. 당신은 도대체 어떤 가치를 제공하고 있는 것입니까?

코드를 검토용으로 제출한다면, 다른 사람들이 시간을 할애할 준비가 되었다고 확신해야 합니다. 초기 검토 과정은 당신의 책임이며, 다른 사람에게 떠넘길 일이 아닙니다.

좋은 에이전틱 엔지니어링 풀 리퀘스트는 다음과 같은 특징을 가집니다:

- 코드가 작동하고, 당신은 그것이 작동한다고 확신합니다. [당신의 일은 작동하는 코드를 전달하는 것](https://simonwillison.net/2024/Mar/4/agentic-engineering-delivering-code/)입니다.
- 변경 사항이 검토자에게 너무 많은 추가적인 인지 부하를 주지 않고 효율적으로 검토될 수 있을 만큼 작습니다. 하나의 큰 PR보다 여러 개의 작은 PR이 낫고, 코딩 에이전트가 Git 작업을 대신해주므로 코드를 별도의 커밋으로 나누는 것은 쉽습니다.
- PR에는 변경 사항을 설명하는 데 도움이 되는 추가적인 맥락이 포함됩니다. 변경 사항이 기여하는 더 높은 수준의 목표는 무엇입니까? 관련 이슈나 사양에 대한 링크가 여기에 유용합니다.
- 에이전트는 설득력 있는 풀 리퀘스트 설명을 작성합니다. 이것들도 당신이 검토해야 합니다! 직접 읽고 검증하지 않은 텍스트를 다른 사람이 읽기를 기대하는 것은 무례한 일입니다.

검토하지 않은 코드를 다른 사람에게 떠넘기기가 얼마나 쉬운지를 감안할 때, 직접 추가 작업을 했다는 증거를 포함하는 것을 권장합니다. 수동으로 테스트한 방법에 대한 메모, 특정 구현 선택에 대한 의견, 심지어 기능이 작동하는 스크린샷과 비디오는 검토자의 시간이 세부 사항을 파헤치는 데 낭비되지 않을 것임을 보여주는 데 _큰_ 도움이 됩니다.
