# 레드/그린 TDD

"**레드/그린 TDD를 사용하라**"는 코딩 에이전트에서 더 나은 결과를 끌어내기 위한 아주 간결하면서도 효과적인 지침입니다.

TDD는 Test Driven Development(테스트 주도 개발)의 약자입니다. 작성하는 모든 코드에 대해, 그 코드가 동작함을 보여주는 자동화 테스트를 함께 갖추는 프로그래밍 스타일입니다.

가장 엄격한 형태의 TDD는 테스트-우선 개발입니다. 먼저 자동화 테스트를 작성하고, 그 테스트가 실패함을 확인한 뒤, 테스트가 통과할 때까지 구현을 반복합니다.

이 방식은 코딩 에이전트와 _환상적으로_ 잘 맞습니다. 코딩 에이전트의 중요한 리스크 중 하나는 동작하지 않는 코드를 만들거나, 필요하지도 않고 결국 사용되지도 않을 코드를 만들 수 있다는 점이며, 둘 다 동시에 일어날 수도 있다는 점입니다.

테스트-우선 개발은 이런 흔한 실수를 둘 다 방지하는 데 도움이 되며, 미래 회귀를 막아주는 견고한 자동화 테스트 스위트도 함께 보장해 줍니다. 프로젝트가 커질수록 새 변경이 기존 기능을 망가뜨릴 가능성도 함께 커집니다. 포괄적인 테스트 스위트는 기능을 계속 정상 동작하게 유지하는 가장 효과적인 방법입니다.

코드를 구현해 테스트를 통과시키기 전에, 테스트가 실제로 실패하는지 반드시 확인하는 것이 중요합니다. 이 단계를 건너뛰면 이미 통과하는 테스트를 만들 위험이 있고, 결과적으로 새 구현을 실제로 검증하지 못하게 됩니다.

이게 바로 "레드/그린"의 의미입니다. 레드 단계에서 테스트 실패를 확인하고, 그다음 그린 단계에서 테스트가 통과함을 확인합니다.

좋은 모델이라면 모두 "red/green TDD"를 "테스트 주도 개발을 사용하고, 테스트를 먼저 작성하고, 테스트를 통과시키는 변경을 구현하기 전에 테스트가 실패함을 확인하라"라는 더 긴 지시의 축약형으로 이해합니다.

예시 프롬프트:

Build a Python function to extract headers from a markdown string. Use red/green TDD.
