# 에이전트 수동 테스트

코딩 에이전트의 결정적인 특징은 자신이 작성한 코드를 _실행할 수 있다_는 것입니다. 이것이 코딩 에이전트를 단순히 코드를 뱉어내고 검증할 방법이 없는 LLM보다 훨씬 더 유용하게 만드는 이유입니다.

LLM이 생성한 코드가 실행되기 전까지는 작동한다고 가정하지 마십시오.

코딩 에이전트는 자신이 생성한 코드가 의도한 대로 작동하는지 확인하거나, 그렇게 될 때까지 해당 코드에 대해 추가로 반복할 수 있는 능력을 가지고 있습니다.

에이전트에게 [단위 테스트를 작성하게 하는 것](https://simonwillison.net/guides/agentic-engineering-patterns/red-green-tdd/), 특히 테스트 우선 TDD를 사용하는 것은 그들이 작성하는 코드를 실행했는지 확인하는 강력한 방법입니다.

하지만 이것만이 유일하게 가치 있는 접근 방식은 아닙니다.

코드가 테스트를 통과했다고 해서 의도한 대로 작동한다는 의미는 아닙니다. 자동화된 테스트를 다루어 본 사람이라면 누구나 테스트는 모두 통과하지만 코드 자체가 명백한 방식으로 실패하는 경우를 본 적이 있을 것입니다. 예를 들어, 시작 시 서버가 충돌하거나, 중요한 UI 요소를 표시하지 못하거나, 테스트가 다루지 못한 일부 세부 사항을 놓치는 경우입니다.

자동화된 테스트는 **수동 테스트**를 대체할 수 없습니다. 저는 릴리스에 포함시키기 전에 제 눈으로 기능이 작동하는 것을 보고 싶습니다.

저는 에이전트에게 코드를 수동으로 테스트하게 하는 것도 가치가 있다는 것을 발견했으며, 이는 자동화된 테스트에서 발견되지 않은 문제를 자주 드러냅니다.

## 에이전트 수동 테스트를 위한 메커니즘

에이전트가 코드 조각을 "수동으로" 테스트하는 방법은 해당 코드가 무엇인지에 따라 다릅니다.

파이썬 라이브러리의 경우 유용한 패턴은 `python -c "... code ..."`입니다. 다른 모듈을 가져오는 코드를 포함하여 파이썬 코드의 문자열(또는 여러 줄 문자열)을 파이썬 인터프리터에 직접 전달할 수 있습니다.

코딩 에이전트는 모두 이 트릭에 익숙하며 때로는 프롬프트 없이도 사용합니다. 하지만 `python -c`를 사용하여 테스트하도록 상기시키는 것이 종종 효과적일 수 있습니다:

Try that new function on some edge cases using `python -c`

다른 언어에도 유사한 메커니즘이 있을 수 있으며, 그렇지 않더라도 에이전트가 데모 파일을 작성한 다음 컴파일하고 실행하는 것은 여전히 빠릅니다. 저는 때때로 나중에 해당 파일이 실수로 리포지토리에 커밋되는 것을 방지하기 위해 순수하게 `/tmp`를 사용하도록 권장합니다.

Write code in `/tmp` to try edge cases of that function and then compile and run it

제 프로젝트 중 다수는 JSON API를 사용하여 웹 애플리케이션을 구축하는 것을 포함합니다. 이러한 경우 에이전트에게 `curl`을 사용하여 실행하도록 지시합니다:

Run a dev server and explore that new JSON API using `curl`

에이전트에게 "탐색"하라고 말하면 종종 새로운 API의 다양한 측면을 시도하게 되어 빠르게 많은 부분을 다룰 수 있습니다.

에이전트가 수동 테스트를 통해 작동하지 않는 것을 발견하면, 저는 그들에게 빨강/초록 TDD로 수정하라고 말하는 것을 좋아합니다. 이렇게 하면 새로운 사례가 영구적인 자동화된 테스트에 의해 다루어지도록 보장됩니다.

## 웹 UI에 브라우저 자동화 사용

수동 테스트 절차를 마련하는 것은 프로젝트에 대화형 웹 UI가 포함된 경우 훨씬 더 가치가 있습니다.

역사적으로 이러한 것들은 코드에서 테스트하기 어려웠지만, 지난 10년 동안 실제 웹 브라우저를 자동화하는 시스템에서 주목할 만한 개선이 있었습니다. 실제 Chrome, Firefox 또는 Safari 브라우저를 애플리케이션에 대해 실행하면 현실적인 환경에서 온갖 종류의 흥미로운 문제를 발견할 수 있습니다.

코딩 에이전트는 이러한 도구를 매우 잘 사용하는 방법을 알고 있습니다.

오늘날 이들 중 가장 강력한 것은 Microsoft에서 개발한 오픈 소스 라이브러리인 **[Playwright](https://playwright.dev/)**입니다. Playwright는 여러 인기 있는 프로그래밍 언어로 바인딩된 모든 기능을 갖춘 API를 제공하며 널리 사용되는 모든 브라우저 엔진을 자동화할 수 있습니다.

단순히 에이전트에게 "Playwright로 테스트해"라고 말하는 것만으로도 충분할 수 있습니다. 그러면 에이전트는 가장 합리적인 언어 바인딩을 선택하거나 Playwright의 [playwright-cli](https://github.com/microsoft/playwright-cli) 도구를 사용할 수 있습니다.

코딩 에이전트는 전용 CLI와 매우 잘 작동합니다. Vercel의 [agent-browser](https://github.com/vercel-labs/agent-browser)는 코딩 에이전트가 사용하도록 특별히 설계된 Playwright 주변의 포괄적인 CLI 래퍼입니다.

제 자신의 프로젝트인 [Rodney](https://github.com/simonw/rodney)도 비슷한 목적을 수행하지만, Chrome DevTools 프로토콜을 사용하여 Chrome 인스턴스를 직접 제어합니다.

다음은 Rodney로 테스트하는 데 사용하는 예제 프롬프트입니다:

Start a dev server and then use `uvx rodney --help` to test the new homepage, look at screenshots to confirm the menu is in the right place

이 프롬프트에는 세 가지 트릭이 있습니다:

*   "use `uvx rodney --help`"라고 말하면 에이전트가 [uvx](https://docs.astral.sh/uv/guides/tools/) 패키지 관리 도구를 통해 `rodney --help`를 실행하게 되며, 이는 Rodney가 처음 호출될 때 자동으로 설치합니다.
*   `rodney --help` 명령은 에이전트가 도구를 이해하고 사용하는 데 필요한 모든 것을 제공하도록 특별히 설계되었습니다. 다음은 [해당 도움말 텍스트](https://github.com/simonw/rodney/blob/main/help.txt)입니다.
*   "look at screenshots"라고 말하면 에이전트가 `rodney screenshot` 명령을 사용해야 함을 암시하고 결과 이미지 파일에 대해 자체 비전 능력을 사용하여 페이지의 시각적 모양을 평가할 수 있음을 상기시킵니다.

짧은 프롬프트에 많은 수동 테스트가 포함되어 있습니다!

Rodney 및 이와 유사한 도구는 로드된 사이트에서 JavaScript 실행, 스크롤, 클릭, 타이핑, 심지어 페이지의 접근성 트리 읽기까지 광범위한 기능을 제공합니다.

다른 형태의 수동 테스트와 마찬가지로, 브라우저 자동화를 통해 발견되고 수정된 문제는 영구적인 자동화된 테스트에도 추가될 수 있습니다.

많은 개발자들은 과거에 자동화된 브라우저 테스트가 불안정하다는 평판 때문에 너무 많은 테스트를 피했습니다. 페이지의 HTML을 아주 약간만 수정해도 실망스러운 테스트 실패의 물결이 발생할 수 있습니다.

코딩 에이전트가 시간이 지남에 따라 이러한 테스트를 유지 관리하게 하면 웹 인터페이스의 디자인 변경에 직면하여 최신 상태로 유지하는 데 관련된 마찰이 크게 줄어듭니다.

## Showboat로 메모하게 하기

에이전트가 코드를 수동으로 테스트하게 하면 추가적인 문제를 발견할 수 있을 뿐만 아니라, 코드를 문서화하고 어떻게 테스트되었는지 보여주는 데 도움이 되는 아티팩트를 생성하는 데에도 사용될 수 있습니다.

저는 에이전트가 _자신의 작업을 보여주도록_ 하는 도전에 매료되었습니다. 데모나 문서화된 실험을 볼 수 있다는 것은 에이전트가 주어진 과제를 포괄적으로 해결했음을 확인하는 매우 유용한 방법입니다.

저는 에이전트 수동 테스트 흐름을 캡처하는 문서를 구축하는 것을 용이하게 하기 위해 [Showboat](https://github.com/simonw/showboat)를 만들었습니다.

다음은 제가 자주 사용하는 프롬프트입니다:

Run `uvx showboat --help` and then create a `notes/api-demo.md` showboat document and use it to test and document that new API.

위의 Rodney와 마찬가지로, `showboat --help` 명령은 에이전트에게 Showboat가 무엇이며 어떻게 사용하는지 가르칩니다. 다음은 [전체 도움말 텍스트](https://github.com/simonw/showboat/blob/main/help.txt)입니다.

세 가지 주요 Showboat 명령은 `note`, `exec`, `image`입니다.

`note`는 Showboat 문서에 마크다운 노트를 추가합니다. `exec`는 명령을 기록한 다음 해당 명령을 실행하고 출력을 기록합니다. `image`는 문서에 이미지를 추가합니다. Rodney를 사용하여 찍은 웹 애플리케이션의 스크린샷에 유용합니다.

`exec` 명령은 이들 중 가장 중요합니다. 왜냐하면 명령과 결과 출력을 함께 캡처하기 때문입니다. 이것은 에이전트가 무엇을 했고 결과가 무엇이었는지 보여주며, 에이전트가 속임수를 써서 _희망했던_ 내용을 문서에 쓰는 것을 막도록 설계되었습니다.

저는 Showboat 패턴이 제 에이전트 세션 동안 달성된 작업을 문서화하는 데 매우 효과적이라는 것을 발견했습니다. 더 넓은 범위의 도구에서 유사한 패턴이 채택되기를 바랍니다.
